1. What is WealthPro?

WealthPro is a digital wealth management platform that connects retail and high-net-worth investors with their relationship managers, dealers, compliance officers, and the back-office administration. The platform handles the entire client lifecycle — from onboarding and KYC, through goal planning and order execution, to compliance checks and periodic reviews.

1.1 The Big Picture

Think of WealthPro as a control room where five different types of people log in and each sees a completely different view of the same data:

👤 CLIENT

The investor. Browses products, places orders, tracks holdings and goals.

🤝 RM (Relationship Manager)

The advisor. Onboards clients, recommends portfolios, monitors goals.

💼 DEALER

The trader. Routes orders to venues, records fills, manages product catalog.

🛡️ COMPLIANCE

The watchdog. Reviews breaches, approves KYC, defines suitability rules.

⚙️ ADMIN

The operator. Manages users, securities, models, and the audit trail.

1.2 Why This Architecture?

WealthPro is split into 11 independent backend services. Each service owns one business domain (Clients, Products, Orders, etc.) and has its own database. This is the microservices pattern — the trade-off is more moving parts, but each team can develop, deploy, and scale a service without touching the others.

1.3 Core Business Flow (One-Sentence Summary)

A client is onboarded by an RM, completes KYC reviewed by Compliance, receives a model portfolio recommendation, places an order that a Dealer executes, holdings get booked into the portfolio, periodic reviews and statements are auto-generated, and the Admin oversees the whole system through audit logs.

2. High-Level Architecture

Every request flows: Browser → API Gateway → Microservice → Database. The Gateway is the only door into the system. Each microservice talks to its own private MySQL database. Services discover each other through Eureka and call each other via Feign (HTTP).

📌 Why one DB per service? If two services shared a database, changing one schema could break the other. Per-service databases give each team full ownership of their data model. The trade-off: cross-service joins become HTTP calls (slower than SQL JOINs but scales horizontally).

5. Workflow #1 — Authentication & Session

Every page except /login and /landing requires a valid JWT token. Here is exactly what happens from the moment a user types their password.

User submits credentials at /login

Frontend (Login.tsx) POSTs {username, password} to /auth/login on the API Gateway.

Gateway looks up the user in gateway_db.users

Uses R2DBC reactive client. Compares the incoming password against the stored BCrypt hash. If mismatch → returns 401.

Gateway issues a signed JWT

Token payload contains: username, roles, clientId (only for CLIENT role). Signed with HS256. Returned to browser as JSON.

Frontend stores token in Zustand auth store

useAuth persists the user object. App router (App.tsx) uses HomeRedirect + getHomeForRole() to send the user to their role-specific dashboard.

Every subsequent API call carries the JWT

Axios interceptor attaches Authorization: Bearer <token>. Gateway's JwtAuthenticationFilter validates the signature and extracts headers username, roles, clientId — these are forwarded to downstream services.

Gateway audits every request

Every successful call is recorded in gateway_db.auditlog with method, path, timestamp, and metadata — visible later in Admin → Audit Log.

Downstream service authorizes by role + ownership

e.g. ClientController rejects a non-RM trying to create a client. A CLIENT can only read their own data — services check that clientId in the URL matches the JWT's clientId.

5.1 ProtectedRoute — How the Frontend Enforces Roles

In App.tsx, every authenticated route is wrapped:

Route prefixAllowed roleExamples
/admin/*ADMINDashboard, Users, Audit, Securities
/rm/*RMClients, Onboard, Recommendations, Goals
/dealer/*DEALEROrder Blotter, Securities, Research Notes

If a CLIENT navigates to /admin/users, ProtectedRoute redirects to /403 (Forbidden). Even if they bypass the frontend, the Gateway + service authorization still rejects them.

7. Workflow #3 — Order Placement & Execution

The heart of the trading system. From "client clicks BUY" to "shares appear in their holdings".

Client browses products

Products.tsx calls GET /api/securities → ProductCatalog returns the list. Client picks a security and quantity, submits the order form.

Order created — status: PLACED

POST /api/orders → order-execution-service inserts into orders table with status=PLACED, fill_quantity=0.

Pre-trade checks run automatically

Four checks: CASH (does client have funds?) SUITABILITY (does this match their risk profile?) EXPOSURE (concentration limits ok?) KYC (is KYC verified?). Each check writes a row to pre_trade_checks with PASS/FAIL.

If any check FAILS → status: REJECTED

A row may also land in analytics_db.compliance_breaches for the Compliance team to review. Client gets a notification with the failure reason.

If all PASS → status: VALIDATED, then ROUTED

Dealer sees the order in OrderBlotter.tsx. Dealer clicks "Route to Venue" → POST /api/orders/{id}/route. routed_venue is set (e.g. SIMULATED_NSE).

Dealer records the fill

POST /api/orders/{id}/fills → row in execution_fills. If full quantity filled → order status FILLED. If partial → PARTIALLY_FILLED.

Allocation creates the holding

Dealer allocates the filled quantity to the client's account → POST /api/orders/{id}/allocations. This triggers PBOR to UPDATE/INSERT in holding (quantity goes up, avg_cost recomputed). Cash is debited from cash_ledger.

Notifications + audit trail

Client gets "Order filled at ₹X" notification. Gateway logs every API hop to auditlog. Analytics service later picks this up for performance computation.

8. Workflow #4 — Compliance & Analytics

Compliance and Analytics work together to keep the platform safe and reportable. These flows happen continuously in the background and surface in dedicated screens.

8.1 Suitability Rules — The Rulebook

Compliance defines rules in SuitabilityRules.tsx. Each rule has:

  • Description: human-readable (e.g. "Conservative clients cannot buy equities")
  • Expression: machine-evaluable string (e.g. client.riskClass != "CONSERVATIVE" || order.assetClass != "EQUITY")
  • Status: ACTIVE / INACTIVE — only ACTIVE rules run on every order

8.2 Breach Detection

When a rule is violated:

  1. order-execution-service writes a FAIL row in pre_trade_checks
  2. analytics-service inserts a row in compliance_breaches with severity (LOW/MEDIUM/HIGH/CRITICAL)
  3. Compliance officer sees it on Breaches.tsx
  4. Officer acknowledges → status: ACKNOWLEDGED. Investigates → CLOSED
  5. A notification is fired to the client and their RM

8.3 Performance & Risk Computation

Daily/monthly batch (analytics-service) computes for every account:

📈 PerformanceRecord

return_percentage vs benchmark_return_percentage over a period (DAILY, MONTHLY, etc.). Surfaces in client Dashboard "Performance vs Benchmark" charts.

📊 RiskMeasure

VOLATILITY, VAR_95, MAX_DRAWDOWN, TRACKING_ERROR. Visible to Compliance in RiskMonitor.tsx for portfolio-level risk monitoring.

9. Workflow #5 — Reviews, Statements & Notifications

9.1 Periodic Reviews

Every quarter / half-year / year, the system schedules a portfolio review per account. The RM conducts the review with their client and updates:

  • review_date — when it actually took place
  • reviewed_by — which RM
  • highlights_json — return%, benchmark, breach_count, recommended action
  • status — SCHEDULED → IN_PROGRESS → COMPLETED

Visible to the client at /me/reviews.

9.2 Statements

Account statements (Monthly / Quarterly / Annual) are auto-generated:

FieldWhat It Stores
summary_jsonOpening balance, closing balance, return%
generated_dateWhen the statement was created
statusPENDING → GENERATED → DELIVERED

9.3 Notifications — The Inbox

Every important event creates a notification row in notification.notification. The user sees them in their bell icon (Notifications.tsx or RmNotifications.tsx).

Categories

Order Compliance CorporateAction Review Each category lights up its own icon.

Status Lifecycle

Unread → Read → Dismissed. User toggles via the UI; the badge counter reflects only Unread.

9.4 Corporate Actions

When Bajaj Finance announces a 1:1 bonus issue, an entry lands in pbor.corporate_action (security_id, ca_type=BONUS, ratio in terms_json, ex_date / record_date / pay_date). Affected clients get a notification, and the holding quantity is updated on pay_date.

12. Key Points of Observation

What to look for to confirm the application is working correctly. These are the "did it really happen?" checkpoints.

✅ Authentication is real

JWT tokens expire. Try editing one in the browser dev tools — every API call returns 401. Role separation is enforced both in frontend (ProtectedRoute) and backend (controller annotations).

✅ Per-service databases

Open MySQL, run SHOW DATABASES; — you'll see 9 separate schemas. Each microservice connects only to its own schema. No cross-service shared tables.

✅ Audit trail is comprehensive

Every API call goes through the gateway. Every gateway request lands in auditlog. The Admin Audit page is a live view of this — try doing anything as any user, then refresh the audit log.

✅ Compliance is data-driven

Suitability rules are stored as data, not hardcoded. A compliance officer can change rules without a code deploy (the evaluation engine is the next iteration — see the gap analysis document).

✅ Notifications fire on events

Place an order → notification appears in client's inbox. Approve KYC → notification fires to the client. Each is a real DB row in notification.notification.

✅ Portfolio Book of Records (PBOR)

After an order fills + allocates, check the PBOR holding table — quantity goes up, avg_cost recomputed. The cash_ledger debits the purchase amount. This is the true source of position truth.

12.1 Things That Surprise First-Time Reviewers

  • The Gateway is the only entry point. Every microservice is locked down — they only accept requests with the auth headers the gateway injects.
  • One DB per service is intentional. Cross-service joins look like Feign HTTP calls in the code (e.g. GoalsAdvisory → ProductCatalogFeignClient).
  • Eureka makes services findable by name, not URL. The code says http://product-catalog/..., not http://localhost:8082/....
  • The frontend is one SPA, but feels like 5 different apps. Same React bundle, same routes, but role-based menu and page filtering create totally different experiences.
  • Every order writes to 3+ databases. orders (order_execution_db), pre_trade_checks (order_execution_db), and after fill — holding (pbor) + cash_ledger (pbor) + notification (notification) + auditlog (gateway_db).

12.2 What to Test if You're Reviewing This Code

  1. Login → verify JWT in dev tools → confirm token contains correct role + clientId
  2. Try cross-role access: as CLIENT, hit /admin/users → should redirect to /403
  3. Place an order → verify rows appear in orders + pre_trade_checks within 1 second
  4. Reject an order via failed suitability → verify breach appears in compliance breaches
  5. Approve a KYC → verify notification fires + client status changes ACTIVE
  6. Run a query on auditlog → verify your every action is logged with timestamp
  7. Stop a service (e.g. ProductCatalog) → verify other services degrade gracefully (or fail loud — both are valid)
  8. Check Eureka dashboard at :8761 → verify all services registered
